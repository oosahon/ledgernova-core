# 11. Risks and Technical Debt

This section systematically identifies, categorizes, and evaluates the known technical risks and accumulated technical debt within the LedgerNova platform. By maintaining an explicit ledger of these systemic liabilities, we ensure that architectural trade-offs remain visible to stakeholders and can be actively managed, prioritized, and mitigated during future development cycles.

The identified risks are categorized into three primary domains: **Business Risks** (affecting compliance and core logic), **Technical Risks** (affecting operability, scale, and system resilience), and **Technical Debt** (suboptimal implementations impacting maintainability).

## 11.1 Business Risks

Business risks encompass systemic gaps or incomplete domain logic that could directly compromise the platform's adherence to regulatory standards (such as the Nigeria Tax Act) or distort financial accuracy.

| Risk                                                  | Severity | Impact                                                                                                                                                                                                                                                                         | Mitigation Strategy                                                                                                                                                            |
| :---------------------------------------------------- | :------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Incomplete Personal Income Tax Aggregation**        | High     | The current `PersonalIncomeTaxPolicy` implementation computes liability on isolated income records, lacking the aggregation required to consolidate multiple income streams. This risks under-calculating the effective tax bracket for users with diversified income sources. | Refactor `taxation-policy.ts` to implement an aggregation pipeline that consolidates all valid income streams within the fiscal period prior to calculating the taxable bands. |
| **Missing Authentication Middleware Implementations** | High     | Unimplemented middleware stubs (`is-optional-authenticated-user.middleware.ts`) may leave certain API endpoints susceptible to unpredictable behavior, unauthorized data exposure, or unhandled server exceptions if anonymously accessed.                                     | Fully implement token extraction and validation using the central JWT service to guarantee the request context is safely populated or explicitly marked anonymous.             |

## 11.2 Technical Risks

Technical risks include architectural bottlenecks, missing access layers, and unhandled edge cases that threaten the runtime stability, performance, and scalability of the platform.

| Risk                                              | Severity | Impact                                                                                                                                                                                                                                       | Mitigation Strategy                                                                                                                                                            |
| :------------------------------------------------ | :------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Unbounded Data Fetching (Category Pagination)** | Medium   | The `GetAllCategoriesUseCase` fetches all system and user domain categories without pagination. As taxonomies expand, these unbounded database queries will invariably cause memory pressure, compute bottlenecks, and degraded API latency. | Introduce mandatory cursor or offset-based pagination constraints at the use case level and enforce them within the underlying Drizzle ORM queries in `category.repo.impl.ts`. |
| **Incomplete Repository Implementations**         | Medium   | Several methods within `CategoryRepoImpl` remain stubbed (`// TODO: implement`). Any downstream domain operations invoking these methods will fail or behave non-deterministically.                                                          | Prioritize the complete implementation of all data access layer (DAL) queries and mutations for the `CategoryRepo`, backed by rigorous integration tests.                      |
| **Unmapped Transaction Typings**                  | Medium   | Incomplete interfaces within `transaction.types.ts` degrade strict TypeScript compilation guarantees. Processing complex or malformed double-entry payloads natively increases the risk of subtle runtime crashes or miscast types.          | Finalize the explicit mapping of all supported transaction structures and enforce boundary validation through unified Zod schemas.                                             |

## 11.3 Technical Debt

Technical debt represents conscious or unconscious implementation shortcuts that significantly increase the cognitive load on developers and the long-term cost of system maintenance, although they may not immediately threaten stability.

| Debt                                              | Impact                                                                                                                                                                                | Refactoring Plan                                                                                                                                                                             |
| :------------------------------------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Hardcoded Ledger Account Code Generation**      | The `LedgerAccount` entity lacks abstractions to dynamically assign deterministic ledger codes based on accounting classifications, relying on manual generation or injection.        | Integrate the internal accounting rule engine (or a domain service factory) to automatically generate standard ledger codes based on the selected subledger mappings during entity creation. |
| **Lack of Localization Support in Value Objects** | Fundamental domain validation errors within `MoneyVO` (`money.vo.ts`) are hardcoded in English, hindering the internationalization strategy and degrading the native user experience. | Decouple error string literals from the domain layer entirely by throwing strongly-typed errors that hook into the global i18n translation dictionary via middleware.                        |
